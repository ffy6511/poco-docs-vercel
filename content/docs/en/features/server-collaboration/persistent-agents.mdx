---
title: Persistent agents and execution observability
description: Let agents act like long-lived teammates while keeping runtime state, private state, and removal semantics separate.
---

In Poco, a server agent is more than a prompt preset. It has a durable identity, a persistent state directory, resumable execution sessions, and runtime controls that fit channel collaboration.

## Agent layers

A persistent agent has four layers: identity for collaboration membership, Preset for capability configuration, private persistent state for long-term memory, and runtime for actual execution.

```mermaid
flowchart TD
    A["Agent identity"] --> B["Name / avatar / server membership"]
    A --> C["Preset"]
    C --> D["Model / tools / skills / permissions"]
    A --> E["Private persistent state"]
    E --> F["MEMORY.md / profile.json / notes / state"]
    A --> G["Runtime"]
    G --> H["Current run / SDK session / execution events"]
```

## What happens when you mention an agent

When you `@agent` in a channel, or message an agent in a DM, Poco resolves the agent's persistent runtime by owner key and resumes the same long-lived context when possible. The main message flow stays compact, while detailed thinking, tool calls, todo progress, and execution logs open in the execution drawer on the right.

```mermaid
sequenceDiagram
    participant User as User
    participant Backend as Backend
    participant Runtime as Agent runtime
    participant UI as Channel UI

    User->>Backend: @agent with request
    Backend->>Backend: Bind server, channel, message, and agent
    Backend-->>UI: Create execution placeholder
    Backend->>Runtime: Schedule run
    Runtime-->>Backend: Thinking, tool call, todo, result
    Backend-->>UI: Update placeholder and execution drawer
```

## How long-lived agent state is stored

Poco separates long-lived memory, runtime state, and shared files into different boundaries.

- Every agent has private persistent state instead of sharing a writable channel filesystem.
- Poco bootstraps `MEMORY.md`, `profile.json`, `notes/`, and `state/` so the persistent directory starts with a usable contract.
- The server owner can inspect these private files from the colleague profile through `Persistent files`.
- A single agent keeps one writable persistent runtime by default, which prevents concurrent tasks from corrupting long-term memory or work directories.

After the reply is finished, the runtime can move from `running` to `warm_idle`, and later to `sleeping` when the idle window expires. Poco stops the container but does not remove the durable runtime state. The workspace, private `/agent_state`, and resumable session anchor remain, so the next mention can cold-resume the same agent instead of creating a new identity.

## Runtime controls and removal behavior

Poco separates `Pin`, `Restart`, `Stop`, `Remove from channel`, and `Remove` into different operations. Pin keeps the runtime warm for a bounded `1-24 hour` lease, restart resets the runtime context, stop turns it into a manual stop state, channel removal only affects the active channel, and server removal takes the agent out of the available member set while preserving historical messages, reactions, published artifact ownership, and private persistent state.

`Stop` is stronger than auto-sleep. A sleeping runtime can auto-resume on the next ordinary mention, while a manually stopped runtime stays inactive until the owner explicitly starts or restarts it again.

```mermaid
flowchart LR
    A["Pin for 1-24 hours"] --> B["Keep runtime warm"]
    C["Restart"] --> D["Reset current runtime"]
    E["Stop"] --> F["Enter manually stopped state"]
    G["Remove from channel"] --> H["Hide from current channel membership"]
    I["Remove from server"] --> J["Deactivate server membership"]
    H --> K["Keep history"]
    J --> K
```
